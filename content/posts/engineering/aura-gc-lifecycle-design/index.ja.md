# Aura 四象限 GC ライフサイクル：メモリロックと自動解放のバランス

![Aura GC ライフサイクル可視化](featured.png)

高性能な AI システムが最も忌み嫌うのは「ガベージコレクション（GC）のジッター」です。24-bit アドレッシングのミリ秒レベルのレスポンスを保証するために、Aura は独自の**四象限メモリライフサイクル管理体系**を設計しました。

## 1. 永続ロック領域：システムの「根」

Aura は、コアとなる 256MB のアドレッシングハッシュテーブルとソウルルールを**永続リソース**として定義しています。
これらのリソースはシステム初期化時に割り当てられ、Rust のメモリロック API（mlock）を使用して物理メモリ内に強制的に常駐させられます。OS によるディスクへのスワップ（Swap）は厳禁されており、$O(1)$ アドレッシングの絶対的な決定性を保証しています。

## 2. タスクレベルの長期リソース

特定の長期タスクのコンテキスト情報は、**タスクレベルリソース**として定義されます。
これらのリソースはタスク開始時に生成され、タスクの最終的なアーカイブ時に破棄されます。これらは SurrealDB に格納され、アクティブな間だけメモリキャッシュにマッピングされます。

## 3. ノードレベルの短期リソース

Matrix が各 24-bit ポインタを実行する際に生成される中間データは、**ノードレベルリソース**に属します。
Rust の **RAII（Resource Acquisition Is Initialization）**メカニズムにより、これらのリソースは Matrix ノードの実行が完了し、変数がスコープを外れた瞬間に即座に物理的に回収されます。バックグラウンドでの GC スキャンは一切不要です。

## 4. 使い捨ての WASM サンドボックス

各スキルプラグインは、独立した WASM インスタンス内で実行されます。スキルの実行が終了すると、インスタンス全体とそれが占有していた線形メモリは瞬時に破棄されます。この「物理的隔離、使い捨て」モデルは、AI スキルプラグインにありがちなメモリリーク問題を根本から断ち切ります。
